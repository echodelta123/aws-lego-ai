# Project 4: Production MLOps Pipeline

## Overview

This project provides the CI/CD system for training and deploying the LightGCN recommendation model. It treats model development like software delivery, with structured steps for testing, evaluation, approval, and deployment.

The pipeline starts from either new training data or code changes and ends with a validated model deployed to production.

Within the wider platform:

* **Model training loop** — Retrains the LightGCN model used by the recommendation system in Project 1
* **Data-driven triggers** — Uses interaction data from the S3 data lake (generated by user activity streams)
* **Controlled deployment** — Only approved models are promoted to production through a blue/green rollout

---

## Architecture

```
Trigger: new data in S3 OR Git commit
        │
        ▼
AWS CodePipeline
        │
        ├── Source
        │     ├── Model training code (CodeCommit)
        │     └── Interaction data (S3 / EventBridge)
        │
        ├── Build & Test (CodeBuild)
        │     ├── Unit tests
        │     ├── Data validation checks
        │     └── Docker image build → ECR
        │
        ├── Train & Evaluate (SageMaker Pipeline)
        │     ├── Data preprocessing (graph construction)
        │     ├── Model training (LightGCN)
        │     ├── Evaluation (accuracy, recall, bias checks)
        │     └── Register model (pending approval)
        │
        ├── Manual Approval
        │     ├── Notification via SNS
        │     ├── Review evaluation report
        │     └── approve / reject decision
        │
        └── Deploy (Blue/Green)
              ├── New endpoint deployed alongside existing one
              ├── 10% canary traffic shift
              ├── CloudWatch monitoring
              └── Automatic rollback if errors increase
```

## SDLC Traceability

Each deployment produces a record linking code, data, and approvals for auditability.

| Field         | Example                         |
| ------------- | ------------------------------- |
| Jira ticket   | AI-247                          |
| Git commit    | a3f9e21                         |
| Training data | s3://data/features/v2024-11-01/ |
| Model package | SageMaker Model Package ARN     |
| Metrics       | accuracy, F1, recall            |
| Approval      | governance sign-off timestamp   |
| Model card    | S3 URI                          |
| Endpoint      | production endpoint name        |

This record is stored in DynamoDB and used as the source of truth for audits.

## Deployment Governance

Every model deployment goes through the same set of checks:

* **Quality thresholds** — Models must meet minimum performance benchmarks before approval
* **Audit trail** — Every deployment is linked to code, data, and evaluation results
* **Pre-release review** — Evaluation reports and model cards are reviewed before production rollout
* **Controlled rollout** — New models are deployed gradually using canary traffic shifts

## Cost and Resource Design

The pipeline is designed to avoid idle infrastructure costs:

* **Ephemeral SageMaker jobs** — Training and evaluation run only when triggered, then shut down
* **Spot instances for training** — Reduces compute cost during model training
* **Serverless CI/CD tools** — CodePipeline and CodeBuild scale to zero when idle
* **No persistent compute** outside of production deployments
